# 프로젝트 개요 및 구현 방안

## 1. 프로젝트 개요

이 프로젝트는 커뮤니티 및 유저 서비스 환경에서 RDB(MySQL), Redis, Elasticsearch를 활용하여 **대용량 데이터(100만 건)의 조회 성능을 다각도로 비교하고 검증**하기 위한 시연용 프로젝트입니다.

조회 방식에 따른 Latency 차이를 측정하여, **서비스 특성(읽기 빈도, 쓰기 빈도, 검색 유형, 정합성 요구 수준)에 따라 어떤 아키텍처를 선택해야 하는지 근거 있는 판단 기준**을 제시하는 것이 최종 목표입니다.

*   **주요 기술 스택:** Spring Boot 4.1 / Java 21, MySQL, Redis, Elasticsearch, Kafka

---

## 2. 성능 비교 대상 (5가지 조회 전략)

| # | 전략 | 핵심 메커니즘 | RDB 의존 여부 |
|---|---|---|---|
| 1 | **RDB 단독** | MySQL LIKE 검색 + 페이징 | ✅ 항상 |
| 2 | **Redis 전략 A** | 검색 조건 전체를 Key로 결과 리스트 통째 캐싱 (Look-aside) | 🔶 Miss 시에만 |
| 3 | **Redis 전략 B** | RDB에서 ID 목록만 페이징 조회 → Redis MGET으로 상세 데이터 조립 | 🔶 항상 (ID 조회용) |
| 4 | **Redis 전략 C** | ZSet으로 정렬/페이징, Hash로 상세 데이터 전부 Redis에서 처리 | ❌ 없음 |
| 5 | **Elasticsearch** | 역인덱스 기반 Full-text 검색 및 페이징 | ❌ 없음 |

### 전략 B vs 전략 C 차별점 명확화

*   **전략 B의 핵심 가치:** 게시글 상세 데이터(본문, 작성자 등)를 Redis에 캐싱함으로써 **데이터 변경 시 해당 건의 캐시만 무효화하면 되는 세밀한 캐시 관리**가 가능합니다. 반면 정렬/페이징은 여전히 RDB에 의존하므로 목록 쿼리 자체의 성능 한계는 남아있습니다.
*   **전략 C의 핵심 가치:** 정렬, 페이징, 상세 데이터 모두 Redis 내부에서 해결하여 **RDB 병목을 완전히 제거**합니다. 단, 구현 복잡도와 Redis 메모리 비용이 가장 높습니다.

## 3. 데이터 동기화 아키텍처 (비동기 이벤트 기반)

이중 쓰기(Dual Write)를 피하고, **Kafka를 활용한 비동기적 최종 일관성(Eventual Consistency)** 방식을 채택합니다.

### 쓰기(C/U/D) 흐름

```
클라이언트 요청
    → [Spring Boot] MySQL에만 저장 (트랜잭션 Commit)
    → 즉시 클라이언트에 성공 응답 반환
    → @TransactionalEventListener(AFTER_COMMIT)
        → KafkaProducer.publish(이벤트)
            → [Kafka Consumer]
                ├─ Redis 동기화 (전략별 로직 분기)
                └─ Elasticsearch 동기화 (Upsert/Delete)
```

### 읽기(R) 흐름

```
클라이언트 조회 요청
    → /api/v1/rdb/**         → MySQL 직접 조회
    → /api/v1/redis/a/**     → Redis(전략A) → Miss 시 MySQL 폴백
    → /api/v1/redis/b/**     → MySQL(ID 목록) + Redis MGET(상세)
    → /api/v1/redis/c/**     → Redis ZSet/Hash에서 완전 조회
    → /api/v1/elastic/**     → Elasticsearch에서 완전 조회
```

### 부분 실패(Partial Failure) 대응 전략

Kafka Consumer에서 Redis/ES 동기화 실패 시 다음과 같이 대응합니다.

*   **Retry:** Consumer는 실패 시 최대 3회 자동 재시도합니다.
*   **Dead Letter Queue (DLQ):** 3회 재시도 후에도 실패한 메시지는 별도의 DLQ Topic(`xxx.DLT`)으로 이동시켜 유실을 방지합니다.
*   **전략 C의 원자성:** ZSet과 Hash 두 자료구조를 동시에 업데이트하는 경우, Redis `MULTI/EXEC` 트랜잭션 또는 Lua Script를 사용하여 부분 적용 상태가 발생하지 않도록 보장합니다.

## 4. 100만 건 데이터 초기 적재 전략

Kafka를 통한 100만 건의 실시간 동기화는 수 시간이 걸릴 수 있어 현실적이지 않습니다. 따라서 **초기 적재와 이후 운영 단계를 분리**합니다.

### Phase 1: 초기 대량 적재 (Bulk Load)
1.  **MySQL Batch Insert:** JPA의 `saveAll()` 대신 JDBC `batchUpdate()`를 사용하여 100만 건을 빠르게 삽입합니다. (Kafka 이벤트 발행 없이 직접 삽입)
2.  **ES Bulk Indexing:** MySQL 적재가 완료된 후, 별도 초기화 API(`POST /api/admin/init/elastic`)를 호출하여 Elasticsearch Bulk API로 100만 건을 한 번에 인덱싱합니다.
3.  **Redis Warm-up:** 별도 Warm-up API(`POST /api/admin/init/redis`)를 호출하여 전략 C의 ZSet/Hash를 사전 적재(Pre-warming)합니다.

### Phase 2: 이후 운영 (증분 동기화)
초기 적재 완료 이후부터 발생하는 데이터 변경(C/U/D)은 기존 Kafka 비동기 동기화 파이프라인으로 처리합니다.

## 5. 검증 테스트 시나리오 (핵심)

> [!IMPORTANT]
> 이 프로젝트의 최종 목표는 단순한 속도 비교가 아니라, **"어떤 서비스 상황에서 어떤 아키텍처를 선택해야 하는가"에 대한 근거를 제시하는 것**입니다. 따라서 시나리오를 서비스 상황별로 구분하여 측정합니다.

### 테스트 공통 조건

*   데이터 규모: 게시글 100만 건 (제목 20~50자, 본문 500~2000자, 태그 포함)
*   모든 시나리오는 동일 환경(Docker, 동일 HW)에서 순차 실행
*   측정 지표: **평균 응답 시간(ms), P95 응답 시간(ms), TPS**
*   성능 측정 도구: Spring 단위 테스트 내 `StopWatch` 활용 + 별도 부하 테스트(선택)

---

### 📌 시나리오 1: 단순 페이징 목록 조회 (Cold Start)

**상황:** 캐시나 인덱스가 완전히 비어있는 초기 상태 (서버 재시작 직후)에서 목록 첫 페이지를 조회합니다.

**의도:** 워밍업이 없는 최악의 조건에서 각 전략의 실제 초기 응답 속도를 비교합니다.

| 요청 | 정렬: 최신순, 페이지: 1, 사이즈: 20 |
|---|---|
| 반복 횟수 | 1회 (Cold Hit 측정) |
| 예상 승자 | RDB, Redis 전략 A(Miss), 전략 B(Miss) 모두 MySQL 의존 → ES와 Redis C가 유리 |

---

### 📌 시나리오 2: 단순 페이징 목록 조회 (Warm-up 후, Repeated Read)

**상황:** 동일한 요청을 100회 반복하여 캐시가 충분히 워밍업된 상태에서의 성능을 비교합니다.

**의도:** "읽기가 압도적으로 많고, 동일 쿼리가 반복되는 서비스(예: 인기 게시판)"에서 어떤 전략이 유리한지 파악합니다.

| 요청 | 정렬: 최신순, 페이지: 1, 사이즈: 20 |
|---|---|
| 반복 횟수 | 100회 (1회 Warm-up + 99회 측정) |
| 예상 승자 | Redis 전략 A (완전 동일 쿼리 캐시 Hit → 최고 속도) |
| 핵심 인사이트 | 전략 A는 Cache Hit 시 가장 빠르지만, 전략 B/C는 Cache Miss도 없으므로 안정적 |

---

### 📌 시나리오 3: 키워드 Full-text 검색

**상황:** 한글 키워드로 게시글 제목 및 본문을 검색합니다. (예: `"스프링 부트 개발"` 검색)

**의도:** "검색 기능이 핵심인 서비스(예: 블로그, 지식인)"에서 각 전략의 한계를 드러냅니다.

| 요청 | 키워드: `"스프링 부트 개발"`, 정렬: 관련도순 |
|---|---|
| 반복 횟수 | 50회 |
| 비고 | Redis 전략들은 Full-text Search를 지원하지 않으므로 RDB의 LIKE 검색으로 폴백됩니다. ES는 한글 형태소 분석기(Nori) 적용 |
| 예상 승자 | **Elasticsearch 압도적 우위** |
| 핵심 인사이트 | 키워드 검색이 핵심 기능이라면 ES는 선택이 아닌 필수임을 수치로 증명 |

---

### 📌 시나리오 4: 고빈도 쓰기 + 읽기 혼합 (Cache Churn)

**상황:** 새 게시글이 지속적으로 작성되는 동안, 동시에 목록 조회 요청을 보냅니다. (쓰기:읽기 = 3:7 비율)

**의도:** "실시간 커뮤니티(예: 실시간 피드, 주식 게시판)"처럼 데이터 변경이 잦은 환경에서 캐시 전략의 효율이 얼마나 무너지는지 확인합니다.

| 요청 | 5초 동안 쓰기 30건 + 읽기 70건 동시 발생 |
|---|---|
| 반복 횟수 | 3세트 |
| 비고 | 전략 A는 쓰기 발생 시마다 캐시가 무효화(Evict)되어 Hit Rate가 폭락할 가능성이 높음 |
| 예상 승자 | Redis 전략 C (쓰기 발생 시 ZSet/Hash를 업데이트하면 되므로 항상 최신 데이터 제공) |
| 핵심 인사이트 | 전략 A의 Hit Rate가 실제로 얼마나 되는지 로그로 측정하여 함께 보고 |

---

### 📌 시나리오 5: 다양한 페이지 무작위 접근 (Random Read)

**상황:** 1페이지~5000페이지 사이의 임의 페이지를 무작위 순서로 조회합니다.

**의도:** "아카이브 서비스나 SNS의 타임라인"처럼 사용자마다 다른 페이지를 접근하는 환경에서 캐시 전략의 한계를 확인합니다.

| 요청 | 랜덤 페이지 1~5000 무작위 조회 |
|---|---|
| 반복 횟수 | 200회 |
| 비고 | 전략 A는 캐시 Key가 페이지마다 달라서 Hit Rate가 극히 낮아짐 |
| 예상 승자 | Redis 전략 C 및 Elasticsearch |
| 핵심 인사이트 | 전략 A는 동일 요청이 반복될 때만 유효하다는 점을 수치로 증명 |

---

### 📌 시나리오 6: 복합 조건 필터링 + 정렬 변경

**상황:** 카테고리 필터 + 특정 날짜 범위 + 조회수 정렬과 같이 복합 조건을 가진 검색을 수행합니다.

**의도:** 서비스가 성숙해질수록 요구되는 복잡한 쿼리에서 각 전략이 어디까지 대응할 수 있는지 확인합니다.

| 요청 | 카테고리: `backend`, 날짜: 최근 30일, 정렬: 조회수 내림차순 |
|---|---|
| 반복 횟수 | 50회 |
| 비고 | Redis 전략 C의 ZSet은 정렬 기준이 1개로 고정되어 있으므로, 정렬 기준 변경 시 별도 ZSet이 필요 |
| 예상 승자 | **Elasticsearch** (복합 필터, 다양한 정렬이 쿼리 하나로 처리됨) |
| 핵심 인사이트 | 서비스의 검색/필터 요구가 복잡해질수록 Redis C의 관리 비용이 기하급수적으로 늘어남 |

---

### 🗂 최종 아키텍처 선택 가이드 (시나리오 결과 기반)

| 서비스 유형 | 추천 전략 | 이유 |
|---|---|---|
| 읽기 압도적, 동일 쿼리 반복 | **Redis 전략 A** | Warm-up 후 Hit Rate 높을 때 가장 빠름 |
| 읽기 많고, 쓰기도 간간이 | **Redis 전략 C** | 캐시 Evict 없이 최신 데이터 유지 가능 |
| 실시간 피드, 잦은 쓰기 | **Redis 전략 C** | 쓰기와 읽기가 충돌 없이 공존 가능 |
| 키워드 검색이 핵심 | **Elasticsearch** | Full-text, 형태소 분석, 오타 허용(Fuzzy) |
| 복합 필터/정렬 다양 | **Elasticsearch** | 유연한 Query DSL로 모든 조건 처리 |
| 소규모, 단순 서비스 | **RDB 단독** | 오버 엔지니어링 방지, 운영 비용 최소화 |
